# Team Review Meeting

PR/merge/배포 전에 가상 팀 리뷰 회의를 진행합니다.

## 프로세스

**반드시 아래 순서를 따르세요. 건너뛰지 마세요.**

### 1. 변경점 수집

```bash
git diff main...HEAD --stat
git diff main...HEAD
git log main..HEAD --oneline
```

변경된 파일, 코드 diff, 커밋 이력을 모두 읽고 이해하세요.

### 2. 팀 리뷰 회의

3명의 팀원이 각자 관점에서 변경점을 검토합니다. **각 팀원은 반드시 구체적인 코드 라인을 언급**해야 합니다.

#### 참석자

| 역할 | 이름 | 검토 관점 |
|------|------|-----------|
| 시니어 개발자 | 현우 | 아키텍처, 설계, 성능, 하위 호환성 (CLI/web 영향) |
| 주니어 개발자 | 도윤 | 가독성, 네이밍, 하드코딩, 기본적인 실수, "이거 왜 이렇게 되어있죠?" |
| QA 엔지니어 | 준혁 | 계산 정확성, 엣지 케이스 (경계값, 0, 음수, 오버플로우), 회귀 가능성 |

#### 회의 형식

각 팀원이 순서대로 발언합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🗣️ [역할] 이름
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[의견]

✅ 승인 / ⚠️ 조건부 승인 / ❌ 반려
사유: ...
```

### 3. 투표 결과 집계

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📊 투표 결과
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✅ 승인: X명
⚠️ 조건부: X명
❌ 반려: X명

최종 결정: [통과 / 토론 필요]
```

### 4. 토론 (반려/조건부가 있을 경우)

❌ 반려 또는 ⚠️ 조건부 의견이 있으면 **즉시 수정하지 말고** 팀 토론을 먼저 진행합니다.

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
💬 토론
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[지적 사항에 대해 팀원들이 자유롭게 의견을 주고받음]
- 다른 팀원이 반론을 제기할 수 있음
- 원래 지적한 팀원이 반론을 수용하거나 재반박할 수 있음
- 대안을 제시할 수 있음
```

토론 후 **최종 합의**를 도출합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 토론 결론
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

수정할 사항: [합의된 수정 목록]
수정 불필요로 합의된 사항: [토론 후 철회된 지적]
추후 과제로 분리: [이번 PR 범위 밖으로 미룰 사항]
```

### 5. 판정 기준

- **전원 ✅ 승인**: 즉시 PR → merge → 배포 진행
- **토론 후 수정 사항 있음**: 수정 후 재리뷰 (`/review` 재실행)
- **토론 후 수정 사항 없음** (지적이 철회됨): PR → merge → 배포 진행

### 6. 통과한 경우

리뷰 통과 = 승인이므로 **PR 생성과 merge까지 자동 진행**한다. 태그만 사용자 확인.

**자동 진행 (리뷰 통과 시):**
1. 브랜치 생성 (없으면)
2. 커밋 & 푸시
3. PR 생성
4. CI 통과 확인
5. Squash merge

**사용자 확인 필요:**
6. 태그 & npm publish — 사용자에게 태그 버전(patch/minor/major) 확인 후 실행. 코드 변경 없는 커밋(docs, chore)은 태그하지 않음.

## 주의사항

- 팀원들은 **아부하지 않습니다**. 진짜 문제가 있으면 반려합니다.
- "사소한 변경이라 괜찮다"는 이유로 리뷰를 생략하지 마세요.
- core는 CLI와 web 모두에 영향을 미칩니다. **하위 호환성**을 반드시 확인하세요.
- 반드시 **실제 코드 diff를 읽고** 리뷰하세요. 추측으로 판단하지 마세요.
- **재리뷰는 최대 2회까지.** 2회 재리뷰 후에도 미해결 사항이 있으면 사용자에게 판단을 위임합니다.
