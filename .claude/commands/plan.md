# Planning Meeting

새 기능 추가, 구조 변경 등 기획이 필요한 작업 전에 가상 팀 기획 회의를 진행합니다.

## 언제 사용하는가

- 새 공개 API 추가 (함수, 타입, 상수)
- 기존 API 동작 변경
- 레벨/티어/EXP 계산 로직 변경
- 타입 구조 변경

## 사용하지 않는 경우

- 버그 수정 (원인이 명확한 경우)
- 단순 리팩터링 (공개 API 변경 없음)
- docs/config 수정
- 질문, 조사, 디버깅

## 프로세스

**반드시 아래 순서를 따르세요.**

### 1. 요청 정리

사용자의 요청을 먼저 정리합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📝 요청 사항
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[사용자가 원하는 것을 1~3문장으로 요약]
```

### 2. 팀 기획 회의

3명의 팀원이 각자 관점에서 기획을 검토하고 의견을 제시합니다.

#### 참석자

| 역할 | 이름 | 검토 관점 |
|------|------|-----------|
| PM | 민수 | 요구사항 정의, 범위 설정, CLI/web 양쪽 소비자 영향 분석 |
| 시니어 개발자 | 현우 | 기술적 실현 가능성, 하위 호환성, API 설계, 순수 함수 원칙 유지 |
| QA 엔지니어 | 준혁 | 경계값, 엣지 케이스, 기존 소비자(CLI/web)에 대한 회귀 영향 |

#### 회의 형식

각 팀원이 순서대로 발언합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🗣️ [역할] 이름
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[의견: 찬성/반대/대안 + 구체적 근거]
```

### 3. 토론

팀원 간 의견이 다르면 자유롭게 토론합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
💬 토론
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[팀원들이 서로 의견을 주고받으며 최선의 방안을 도출]
```

### 4. 결론

토론 결과를 정리합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 기획 결론
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

구현할 내용:
- [합의된 구현 사항 목록]

구현하지 않을 내용:
- [논의 후 제외하기로 한 사항 + 이유]

추후 과제:
- [이번에는 안 하지만 나중에 할 사항]

예상 영향 범위:
- [변경될 파일/함수 목록]
- [CLI/web 소비자에 필요한 업데이트 여부]
```

### 5. 사용자 확인

결론을 사용자에게 보여주고 **승인을 받은 후** 구현을 시작합니다. 사용자가 수정을 원하면 토론을 재개합니다.

## 주의사항

- 팀원들은 **무조건 찬성하지 않습니다**. 하위 호환성이 깨지거나 순수 함수 원칙에 어긋나면 반대합니다.
- PM은 **범위를 제한**하는 역할입니다. core에 I/O나 의존성이 들어오는 것을 경계합니다.
- 시니어 개발자는 **API 설계와 하위 호환성**을 반드시 확인합니다.
- core 변경은 CLI와 web 모두에 영향을 줍니다. 소비자 체크 예시:
  - `getLevelInfo` 반환값 구조가 바뀌면 CLI 출력 포맷과 web 대시보드 렌더링 모두 깨지는가?
  - 새 함수를 `index.ts`에 export하면 CLI/web에서 바로 사용 가능한가, 추가 작업이 필요한가?
  - 티어/이모지 변경 시 CLI 터미널과 web 브라우저에서 모두 정상 렌더링되는가?
- 기획 단계에서 코드를 작성하지 마세요. 합의 후 구현합니다.
